# Docker Setup for Key Service

This directory contains all Docker-related files for the key-service application.

## Files

- `Dockerfile` - Production-ready multi-stage build
- `Dockerfile.dev` - Development build with hot reloading
- `docker-compose.yml` - Production compose configuration with PostgreSQL mTLS
- `docker-compose.test.yml` - Test database (plain TCP, no TLS)
- `postgres/entrypoint.sh` - Copies TLS material with postgres-readable permissions before server start
- `postgres/pg_hba.conf` - Requires `hostssl` + client certificate verification for TCP connections
- `certs/postgres/` - Generated TLS PKI (gitignored; see `npm run docker:certs`)
- `.dockerignore` - Files to exclude from Docker build context
- `signing-key` - **Local only** (gitignored). Generated by `npm run docker:signing-key` for Docker Compose volume mount.

## Quick Start

### Signing key

Docker Compose requires a signing key file mounted at `/run/secrets/signing-key`. See the main [README](../README.md#signing-key-required-for-docker-compose) for local dev vs production guidance.

```bash
# Local dev: generate gitignored docker/signing-key
npm run docker:signing-key

# Production: create your own secret and point the compose volume at it
openssl rand -base64 64 > /secure/path/signing-key
chmod 600 /secure/path/signing-key
```

### PostgreSQL mTLS

The production compose stack uses mutual TLS between key-service and PostgreSQL.

```bash
# Generate local CA + server + client certificates (gitignored under docker/certs/postgres/)
npm run docker:certs

# Start full stack (runs docker:certs and docker:signing-key automatically)
npm run docker:up
```

Certificates are valid for 825 days by default (`POSTGRES_TLS_DAYS` to override). Regenerate with:

```bash
npm run docker:certs -- --force
```

After regenerating TLS or signing keys, reset local database state if connections fail or encrypted keys cannot be decrypted:

```bash
docker compose -f docker/docker-compose.yml down -v
npm run docker:up
```

**Host dev** (`npm run dev` on the host, postgres in Docker): start postgres only, export TLS env vars — see main [README](../README.md#manual-database-setup).

**Permission note:** Generated `client.key` and `docker/signing-key` use mode `644` so the distroless `nonroot` user (uid 65532) can read bind-mounted secrets. Files are gitignored and for local use only. Kubernetes mounts use `defaultMode` on secrets instead.

### Manual Docker commands

```bash
# Production (use --pull so distroless/OpenSSL layers stay current)
docker build --pull -t key-service:latest -f docker/Dockerfile .

# Local stack (generates TLS certs and signing key if missing)
npm run docker:up

# Or step by step
npm run docker:certs
npm run docker:signing-key
docker compose -f docker/docker-compose.yml up -d

# Development
docker build -t key-service:dev -f docker/Dockerfile.dev .
docker-compose -f docker/docker-compose.dev.yml up --build
```

## Docker Images

### Production Image (`key-service:latest`)

- Multi-stage build for optimized size
- Runtime: `gcr.io/distroless/nodejs24-debian12:nonroot` (Alpine for build stages); rootless UID/GID **65532** (`nonroot`)
- Refresh the distroless base (including OpenSSL) with `docker build --pull` when rebuilding
- Health checks included
- Production dependencies only (no `tsx` / `esbuild` in the runtime image)

### Development Image (`key-service:dev`)

- Includes all development dependencies
- Hot reloading support
- Volume mounting for live code changes
- Suitable for local development

## Security Features

- Non-root user execution (distroless `nonroot`, UID/GID 65532)
- Minimal production runtime (distroless Debian 12 + Node); build stages use Alpine
- Production-only dependencies in final image (dev tooling such as `tsx` excluded)
- Proper file permissions

- Development dependencies included
